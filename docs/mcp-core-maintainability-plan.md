# План повышения понятности и поддерживаемости MCP-ядра

## Контекст
План охватывает ядро MCP-решения, которое сейчас сосредоточено в модулях:
- `exts/client-mcp/src/CommonModules/Мсп_Сервер/Module.bsl`
- `exts/client-mcp/src/CommonModules/Мсп_ПротоколКлиент/Module.bsl`
- `exts/client-mcp/src/CommonModules/Мсп_ТранспортКлиент/Module.bsl`
- `exts/client-mcp/src/CommonModules/Мсп_РеестрКлиент/Module.bsl`
- `exts/client-mcp/src/CommonModules/Мсп_Ошибки/Module.bsl`
- `tests/src/CommonModules/ОМ_ПротоколКлиент/Module.bsl`
- `tests/src/CommonModules/ОМ_Сервер/Module.bsl`

Ключевые проблемы текущего состояния:
- правила task flow распределены между протоколом и транспортом;
- внутренние контракты представлены неявными структурами со строковыми ключами;
- терминология вокруг `Асинхронный` частично устарела;
- защитная сетка тестов вокруг публичного API протокола частично сломана;
- `Мсп_ТранспортКлиент` совмещает слишком много обязанностей.

## Главная цель
Сделать ядро MCP-решения понятным для нового разработчика и предсказуемым для сопровождения без широкого рефакторинга за один шаг.

## Подцели
- восстановить доверие к тестам вокруг основного контракта;
- привести код, комментарии и документацию к одной терминологии;
- сделать внутренние структуры и переходы между слоями явными;
- сократить связность между транспортом, протоколом и facade-модулем;
- подготовить кодовую базу к дальнейшим небольшим изменениям без каскадных правок.

## Не цели
- не менять функциональное поведение MCP без отдельного решения;
- не делать массовое переименование полей и модулей в один проход;
- не выполнять широкую косметическую чистку или форматирование;
- не перерабатывать все инструменты и ресурсы вне ядра текущего контракта.

## Ограничения выполнения
- минимальные диффы на каждый шаг;
- один логический шаг = один небольшой scoped change;
- каждый этап должен завершаться проверкой;
- перенос ответственности делать только после того, как тесты фиксируют текущее поведение.

## Принципы выполнения
- сначала стабилизировать контур, потом упрощать архитектуру;
- не смешивать изменение поведения и улучшение структуры в одном этапе;
- каждую новую внутреннюю структуру оформлять через конструктор или явный helper;
- снижать количество двусторонних зависимостей между модулями;
- сохранять facade `Мсп_Сервер` как основной вход для прикладного кода.

## Этапы

### Этап 1. Стабилизация тестового контура
- Статус: `todo`
- Цель:
  вернуть надёжную обратную связь по основному MCP-контракту перед архитектурными изменениями.
- Основная проблема:
  часть сценариев `ОМ_ПротоколКлиент` не проверяет реальное поведение из-за вызова несуществующего метода `ВызоватьИнструмент`.
- Шаги:
  1. Исправить вызовы публичного API в `tests/src/CommonModules/ОМ_ПротоколКлиент/Module.bsl`.
  2. Убедиться, что тесты покрывают оба режима работы async-инструмента:
     sync с немедленным результатом и task-based вызов.
  3. Добавить недостающие негативные сценарии на routing task error и на запрет task-mode для sync-инструмента, если они не покрыты после исправления.
- Затрагиваемые файлы:
  - `tests/src/CommonModules/ОМ_ПротоколКлиент/Module.bsl`
  - при необходимости `tests/src/CommonModules/ОМ_Сервер/Module.bsl`
- Проверка:
  - `v8-test-runner ... test module ОМ_ПротоколКлиент`
  - `v8-test-runner ... test module ОМ_Сервер`
- Критерии готовности:
  - тесты проходят стабильно;
  - ключевые сценарии task flow проверяют именно реальное публичное API.
- Риски:
  - возможны скрытые несовпадения между ожидаемым и фактическим контрактом после восстановления тестов.

### Этап 2. Выравнивание терминологии и комментариев
- Статус: `todo`
- Цель:
  убрать расхождение между кодом, комментариями и текущей моделью task flow.
- Основная проблема:
  поле `Асинхронный` и ряд комментариев всё ещё описывают старую streaming-модель, хотя код уже использует task-based semantics.
- Шаги:
  1. Обновить комментарии в `Мсп_Сервер`, `Мсп_ПротоколКлиент`, `Мсп_РеестрКлиент`.
  2. Зафиксировать в комментариях фактическую семантику:
     `Асинхронный` означает поддержку task-based выполнения, а не потоковый режим.
  3. Проверить `docs/extending.md` и соседние материалы на остатки старой терминологии.
  4. При необходимости добавить короткое пояснение о переходной терминологии, чтобы не делать массовый rename на этом этапе.
- Затрагиваемые файлы:
  - `exts/client-mcp/src/CommonModules/Мсп_Сервер/Module.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_ПротоколКлиент/Module.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_РеестрКлиент/Module.bsl`
  - `docs/extending.md`
- Проверка:
  - ручной grep по `потоков`, `stream`, `responseMode` в ядре и docs;
  - повторный запуск профильных модулей тестов.
- Критерии готовности:
  - новый разработчик читает комментарии и получает актуальную модель;
  - в ядре не остаётся явных описаний старого streaming-подхода.
- Риски:
  - без careful wording можно создать двусмысленность между текущим именем поля и его новой трактовкой.

### Этап 3. Явное оформление внутренних контрактов
- Статус: `todo`
- Цель:
  сократить количество неявных структур и строковых ключей в горячих участках.
- Основная проблема:
  `Запрос`, `Контекст`, `ПараметрыВызова`, `ЗавершениеЗадачи`, MCP-описания формируются вручную в нескольких местах.
- Шаги:
  1. Выделить конструкторы для внутренних DTO:
     - контекст входящего tool call;
     - параметры вызова инструмента;
     - результат завершения задачи;
     - описание MCP tool/resource/prompt для транспорта.
  2. Постепенно заменить ручные `Новый Структура(...)` в ядре на эти конструкторы.
  3. Сгруппировать правила нормализации рядом с соответствующим контрактом, а не размазывать их по нескольким процедурам.
- Вариант реализации:
  - либо расширить существующие модули минимальными конструкторами;
  - либо ввести узкий служебный модуль контрактов, если это уменьшит дублирование без создания лишнего слоя.
- Затрагиваемые файлы:
  - `exts/client-mcp/src/CommonModules/Мсп_ПротоколКлиент/Module.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_ТранспортКлиент/Module.bsl`
  - при необходимости новый узкий служебный модуль контрактов
- Проверка:
  - модульные тесты на конструкторы и на сохранение текущего поведения;
  - просмотр diff на сокращение ручной сборки структур в hot path.
- Критерии готовности:
  - ключевые структуры создаются через единообразные функции;
  - изменение поля в контракте требует правки в одном очевидном месте.
- Риски:
  - слишком раннее введение нового модуля может увеличить сложность вместо упрощения.

### Этап 4. Разделение обязанностей между протоколом и транспортом
- Статус: `todo`
- Цель:
  разорвать двустороннюю зависимость между `Мсп_ПротоколКлиент` и `Мсп_ТранспортКлиент`.
- Основная проблема:
  сейчас протокол знает о task lifecycle в транспорте, а транспорт знает о деталях подготовки протокольных ответов.
- Целевое распределение ответственности:
  - `Мсп_ПротоколКлиент`:
    отвечает за валидацию, нормализацию, формирование MCP-ответов и маппинг ошибок.
  - `Мсп_ТранспортКлиент`:
    отвечает за работу с компонентой, доставку входящих/исходящих сообщений и lifecycle сервера.
  - отдельный orchestration layer:
    отвечает за выполнение task flow и routing результата между слоями.
- Шаги:
  1. Зафиксировать, какие функции должны остаться в протоколе, а какие нет.
  2. Перенести orchestration task flow из протокола в отдельную точку координации.
  3. Оставить транспорту только вызов координирующего слоя и отправку ответов в компоненту.
  4. Убрать прямые вызовы `Мсп_ТранспортКлиент.*` из `Мсп_ПротоколКлиент`.
- Затрагиваемые файлы:
  - `exts/client-mcp/src/CommonModules/Мсп_ПротоколКлиент/Module.bsl`
  - `exts/client-mcp/src/CommonModules/Мсп_ТранспортКлиент/Module.bsl`
  - возможно новый coordination module
- Проверка:
  - `rg` по двусторонним зависимостям между модулями;
  - прохождение профильных тестов;
  - ручной просмотр call graph.
- Критерии готовности:
  - зависимость становится однонаправленной;
  - task flow читается как отдельный сценарий, а не как набор встречных вызовов.
- Риски:
  - этап может затронуть несколько hot path сразу, поэтому его нужно дробить на маленькие подшаги.

### Этап 5. Декомпозиция `Мсп_ТранспортКлиент`
- Статус: `todo`
- Цель:
  уменьшить размер и когнитивную нагрузку транспортного модуля.
- Основная проблема:
  модуль одновременно управляет компонентой, состоянием сервера, task lifecycle, sync реестров и разбором входящих событий.
- Предлагаемая декомпозиция:
  - модуль lifecycle сервера и компоненты;
  - модуль task transport API;
  - модуль обработки входящих MCP-событий;
  - модуль синхронизации реестров с компонентой.
- Шаги:
  1. Выделить самый стабильный повторяющийся блок:
     task completion/status/progress API.
  2. Затем вынести sync реестров.
  3. Затем выделить parsing/dispatch входящих событий.
  4. Оставить в `Мсп_ТранспортКлиент` только facade или координацию верхнего уровня.
- Затрагиваемые файлы:
  - `exts/client-mcp/src/CommonModules/Мсп_ТранспортКлиент/Module.bsl`
  - новые узкие служебные модули транспорта
- Проверка:
  - smoke build;
  - профильные тесты;
  - ручной просмотр размера и числа обязанностей в модуле.
- Критерии готовности:
  - у каждого нового модуля одна понятная роль;
  - новый разработчик может читать transport path по частям.
- Риски:
  - без этапа 4 эта декомпозиция даст только косметический выигрыш.

### Этап 6. Решение по переименованию `Асинхронный`
- Статус: `todo`
- Цель:
  принять осознанное решение, оставляем ли текущее имя как совместимый термин или вводим новое.
- Основная проблема:
  текущее имя не полностью отражает task-based смысл, но его массовое переименование может дать слишком широкий дифф.
- Варианты:
  1. Оставить `Асинхронный`, но задокументировать как технический термин внутри проекта.
  2. Ввести новое поле или accessor с обратной совместимостью.
  3. Сделать controlled rename отдельной задачей после стабилизации архитектуры.
- Шаги:
  1. Оценить, где поле используется в коде и тестах.
  2. Выбрать вариант с минимальным риском для API и поддержки.
  3. Если выбран rename, выполнить его отдельной итерацией с полной прогонкой тестов.
- Проверка:
  - grep по использованию поля;
  - ручная проверка docs и примеров;
  - профильные тесты.
- Критерии готовности:
  - термин не вводит в заблуждение;
  - команда понимает, какой термин является canonical.
- Риски:
  - rename без стабилизации предыдущих этапов создаст лишний шум в истории и ревью.

## Формат выполнения
- каждый этап выполнять отдельным scoped commit;
- перед началом этапа фиксировать точную цель и список файлов;
- после завершения этапа фиксировать:
  - что изменено;
  - какие команды проверки выполнены;
  - что осталось вне текущего scope.

## Минимальный Definition of Done для каждого этапа
- изменения не выходят за пределы заявленного scope;
- тесты и/или проверки для этапа выполнены;
- комментарии и имена не противоречат фактическому поведению;
- следующий разработчик может понять новый слой без чтения всего ядра целиком.
