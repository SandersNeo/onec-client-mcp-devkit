# ADR-0002: Запускать MCP-сервер из командной строки клиента 1С

- Статус: accepted
- Дата: 2026-04-27
- Связанная задача: [GitHub issue #3](https://github.com/1c-neurofish/onec-client-mcp-devkit/issues/3)
- Связанные документы: [ADR-0001](0001-client-mcp-provider-architecture.md), [Инварианты решения](../solution-invariants.md)

## Контекст

Сейчас MCP-сервер внутри `exts/client-mcp` можно запустить программно через `Мсп_Сервер.ЗапуститьСервер(...)` или вручную через форму `Мсп_УправлениеMCP`. Это достаточно для интерактивной работы, но неудобно для сценариев, где клиент 1С стартует как часть внешнего workflow и MCP endpoint должен стать доступен сразу после запуска приложения.

В 1С для передачи команды при старте используется параметр командной строки `/C`, доступный в приложении как `ПараметрЗапуска`. При этом действующие инварианты требуют, чтобы модуль приложения `exts/client-mcp/src/Configuration/ManagedApplicationModule.bsl` содержал только глобальную переменную состояния и обработчики событий приложения. Значит, разбор команды запуска и подготовка параметров MCP-сервера должны жить в общем модуле, а обработчик `ПриНачалеРаботыСистемы` может только делегировать запуск.

## Решение

Добавить запуск MCP-сервера из параметра командной строки `/C`.

Поддерживаемый контракт:

```text
/C"runMcp"
/C"runMcp=/tmp/mcp-conf.json"
/C"runMcp=/tmp/mcp-conf.json;mcpPort=123"
```

- Ключ запуска `runMcp` регистронезависимый.
- Если значение `runMcp` заполнено, оно трактуется как путь к внешнему JSON-файлу конфигурации.
- Параметры из JSON-файла применяются первыми.
- Параметры из строки `/C` имеют приоритет над параметрами из JSON-файла.
- В JSON-файле параметры называются без префикса MCP, например `port` и `timeout`.
- В строке `/C` параметры MCP-сервера называются с префиксом `mcp`, например `mcpPort`.
- Порт по умолчанию остается `8080`, как в форме управления MCP, если порт не задан ни в JSON-файле, ни в `/C`.

Минимальный JSON-конфиг:

```json
{
  "port": 8080,
  "timeout": 900
}
```

Не цели:

- Не добавлять автопоиск конфигурационных файлов без явного пути в `runMcp`.
- Не переносить разбор параметров или бизнес-логику запуска в модуль приложения.
- Не менять архитектуру провайдеров, транспорт, протокол MCP, execution mode или envelope ответов.
- Не добавлять скрытые адаптеры публичного API; допустимая регистронезависимость ограничена ключами параметра запуска.

## Последствия

- Клиент 1С можно запускать сразу с MCP-сервером из внешних workflow и UAT-сценариев.
- Появляется новый публичный контракт запуска через `/C`, поэтому его нужно покрыть тестами и документировать вместе с ядром `client-mcp`.
- Разделение имен `port` в JSON и `mcpPort` в `/C` сохраняет конфиг MCP компактным, но требует явного маппинга при чтении параметров запуска.
- `timeout` остается параметром JSON-конфига и не получает `/C` override без отдельного решения.
- Ошибки в JSON-файле, невалидный путь или некорректный порт должны приводить к явному отказу запуска, а не к молчаливому использованию значения по умолчанию.

## Рассмотренные альтернативы

### 1. Запускать сервер только через форму управления

Этот вариант уже работает, но не подходит для автоматизированного старта клиента 1С: внешний процесс не получает MCP endpoint без ручного действия пользователя.

### 2. Использовать только плоские параметры `/C` без JSON-файла

Плоские параметры проще разобрать, но плохо масштабируются при расширении настроек сервера. JSON-файл позволяет заранее хранить повторяемую конфигурацию, а `/C` использовать только для точечных override.

### 3. Передавать JSON прямо в `/C`

Такой формат уменьшает число файлов, но хуже читается, сложнее экранируется в командной строке 1С и менее удобен для повторного использования в UAT.

## План реализации

- Затрагиваемые области:
  - `exts/client-mcp/src/Configuration/ManagedApplicationModule.bsl`
  - новый или существующий общий модуль в `exts/client-mcp/src/CommonModules/` для разбора параметра запуска MCP
  - `exts/client-mcp/src/CommonModules/Мсп_Сервер/Module.bsl`, если потребуется небольшой публичный helper для запуска по подготовленной структуре параметров
  - `tests/src/CommonModules/ОМ_Сервер/Module.bsl` или отдельный `ОМ_*` модуль для тестов разбора и приоритета параметров
  - пользовательская документация, где описывается запуск `client-mcp`

- Правила внесения изменений:
  - В обработчике `ПриНачалеРаботыСистемы` оставить только вызовы общих модулей: существующую `Мсп_Сервер.Инициализировать()` и делегирование command-line startup.
  - Разбор `ПараметрЗапуска` выполнять в общем модуле `exts/client-mcp`, не в модуле приложения.
  - Сначала прочитать JSON-конфиг из значения `runMcp`, если значение заполнено, затем применить override из `/C`.
  - Для JSON использовать ключ `port`; для `/C` использовать ключ `mcpPort`.
  - Проверять порт явно: значение должно быть числом больше нуля. Невалидное значение не заменять silently на `8080`.
  - Не добавлять fallback-алиасы для `StartMCPServer`, `mcpServer`, `Port` или других имен без отдельного решения.
  - Не менять `Мсп_Транспорт*`, `Мсп_ПротоколКлиент` и provider discovery, если реализация не требует изменения транспортного контракта.

## Проверка

- [ ] `/C"runMcp"` запускает MCP-сервер на порту `8080`.
- [ ] Ключ запуска `runMcp` распознается регистронезависимо.
- [ ] `/C"runMcp=/tmp/mcp-conf.json"` читает JSON-файл и использует поле `port`.
- [ ] `/C"runMcp=/tmp/mcp-conf.json;mcpPort=123"` запускает сервер на порту `123`, даже если в JSON указан другой `port`.
- [ ] Невалидный `mcpPort`, невалидный `port` в JSON или отсутствующий JSON-файл дают явную ошибку запуска, без молчаливого fallback на порт по умолчанию.
- [ ] `ManagedApplicationModule.bsl` по-прежнему содержит только глобальную переменную состояния и обработчики событий приложения.
- [ ] Тесты покрывают разбор `/C`, JSON-конфиг и приоритет override из командной строки.
