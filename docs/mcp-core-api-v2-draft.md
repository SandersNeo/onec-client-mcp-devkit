# Черновик API v2 для ядра MCP

## Scope

Документ описывает целевой публичный API ядра MCP в расширении `client-mcp`.

В scope входят:
- запуск и остановка MCP-сервера;
- регистрация инструментов, ресурсов, шаблонов ресурсов и промптов;
- контракт обработчиков;
- task-based выполнение;
- контракт результатов и ошибок;
- структурный конфиг сервера.

В scope не входят:
- API test client;
- UI-URI и UI-tools;
- внутренние детали транспорта и протокольного роутинга.

## Цели

- отделить публичный API от внутренних модулей;
- убрать служебные поля из descriptor-контрактов;
- сделать контракты вызова, результата и ошибки явными;
- оставить только два режима выполнения: `sync` и `task`;
- сохранить поддержку внешней регистрации из внешних обработок.

## Публичные модули

Публичными считаются только:
- `Мсп_Сервер`
- `Мсп_Схема`
- `Мсп_Результаты`
- `Мсп_Ошибки`

Внутренними считаются:
- `Мсп_ПротоколСлужебный`
- `Мсп_ТранспортСлужебный`
- `Мсп_ТранспортСерверСлужебный`
- `Мсп_ТранспортСобытияСлужебный`
- `Мсп_ТранспортРеестрыСлужебный`
- `Мсп_РеестрСлужебный`
- `Мсп_СостояниеСлужебный`
- другие служебные модули ядра

Для внутренних модулей используется суффикс `Служебный`.

Прикладной код и внешние обработки не должны вызывать внутренние модули напрямую.

## Модель регистрации

Регистрация выполняется только через статические методы `Мсп_Сервер`.

Источник регистрации не входит в descriptor. Он передается отдельным параметром в момент регистрации и хранится в реестре как служебная metadata.

Это дает:
- autodiscovery-провайдерам возможность регистрировать сущности без ручного заполнения поля владельца;
- внешним обработкам возможность регистрировать сущности "снаружи";
- снятие регистрации пакетом по строковому идентификатору источника.

### Правила

- `ИсточникРегистрации` является строкой.
- риск коллизии строковых источников принимается.
- `ИсточникРегистрации` может быть пустым.
- runtime при autodiscovery сам подставляет имя источника.
- внешний код передает `ИсточникРегистрации` явно.

### Публичные методы регистрации

```bsl
Процедура ЗарегистрироватьИнструмент(ОписаниеИнструмента, ИсточникРегистрации = "") Экспорт
Процедура ЗарегистрироватьРесурс(ОписаниеРесурса, ИсточникРегистрации = "") Экспорт
Процедура ЗарегистрироватьШаблонРесурса(ОписаниеШаблонаРесурса, ИсточникРегистрации = "") Экспорт
Процедура ЗарегистрироватьПромпт(ОписаниеПромпта, ИсточникРегистрации = "") Экспорт
Процедура ОтменитьРегистрацию(ИсточникРегистрации) Экспорт
```

## Descriptor-контракты

### Инструмент

```bsl
Функция НовыйИнструмент(
	Имя,
	Описание,
	СхемаПараметров,
	Обработчик) Экспорт
```

Описание инструмента:
- `Имя`
- `Описание`
- `СхемаПараметров`
- `Обработчик`
- `РежимВыполнения`

`РежимВыполнения` допускает только:
- `sync`
- `task`

### Ресурс

```bsl
Функция НовыйРесурс(
	URI,
	Имя,
	Описание,
	MimeType,
	Обработчик) Экспорт
```

Описание ресурса:
- `URI`
- `Имя`
- `Описание`
- `MimeType`
- `Обработчик`

### Шаблон ресурса

```bsl
Функция НовыйШаблонРесурса(
	URIШаблон,
	Имя,
	Описание,
	MimeType,
	Обработчик) Экспорт
```

Описание шаблона ресурса:
- `URIШаблон`
- `Имя`
- `Описание`
- `MimeType`
- `Обработчик`

### Промпт

```bsl
Функция НовыйПромпт(
	Имя,
	Описание,
	Аргументы,
	Обработчик) Экспорт
```

Описание промпта:
- `Имя`
- `Описание`
- `Аргументы`
- `Обработчик`

## Контракт обработчика

Каждый обработчик получает один объект вызова.

```bsl
Функция Обработать(Вызов) Экспорт
```

Вызов содержит:
- `Аргументы` - аргументы вызова;
- `Сессия` - сведения о MCP-сессии;
- `Метаданные` - технические метаданные вызова;
- `ИдентификаторЗадачи` - строка task-based вызова или пустое значение.

### Сессия

`Сессия` содержит:
- `Идентификатор`
- `ИнформацияОКлиенте`
- `ВозможностиКлиента`

### Метаданные

`Метаданные` содержит:
- `ИдентификаторЗапроса`
- `ProgressToken`
- нормализованный `_meta`

`raw` в публичный контракт не входит.

## Модель выполнения

В ядре поддерживаются только два режима.

### `sync`

Обработчик обязан вернуть результат в рамках текущего вызова.

Для `sync`-инструмента:
- `ИдентификаторЗадачи` не передается;
- завершение через task API не используется.

### `task`

Обработчик может не возвращать итоговый результат сразу.

Для `task`-инструмента:
- в вызов передается `ИдентификаторЗадачи`;
- управление задачей идет только через отдельные методы `Мсп_Сервер`.

Режим `sync_or_task` в API v2 не поддерживается.

## Task API

Task API остается набором статических методов `Мсп_Сервер`.

```bsl
Процедура УстановитьСтатусЗадачи(ИдентификаторЗадачи, Статус, Сообщение = "") Экспорт
Процедура СообщитьПрогрессЗадачи(ИдентификаторЗадачи, Прогресс, Всего = Неопределено, Сообщение = "") Экспорт
Процедура ЗавершитьЗадачуУспешно(ИдентификаторЗадачи, Результат) Экспорт
Процедура ЗавершитьЗадачуСОшибкой(ИдентификаторЗадачи, КодОшибки, Ошибка) Экспорт
Функция ЗадачаОтменена(ИдентификаторЗадачи) Экспорт
```

Это решение согласовано с ограничениями 1С:
- без object-handle;
- без методов в структурах;
- со статическими методами и глобальным состоянием.

## Результаты

Результаты создаются через `Мсп_Результаты`.

```bsl
Функция Текст(ТекстРезультата) Экспорт
Функция JSON(Данные) Экспорт
Функция Ресурс(URI, Данные, MimeType = "application/json") Экспорт
Функция Промпт(Сообщения) Экспорт
```

Правила:
- обработчик не возвращает произвольный тип с неявной нормализацией;
- публичный API результата должен быть явным;
- сериализация выполняется внутри builders результата.

## Ошибки

Ошибки создаются через `Мсп_Ошибки`.

```bsl
Функция НекорректныеПараметры(Сообщение) Экспорт
Функция НеНайдено(Сообщение) Экспорт
Функция Внутренняя(Сообщение) Экспорт
Функция ДоступЗапрещен(Сообщение) Экспорт
Функция СбойВыполнения(Сообщение) Экспорт
```

Правила:
- обработчик возвращает явную ошибку API;
- маппинг в JSON-RPC и task-body является внутренней обязанностью ядра;
- прикладной код не работает напрямую с JSON-RPC envelope.

## Конфиг сервера

Конфиг сервера должен задаваться структурой, а не строкой JSON.

```bsl
Функция НовыйПараметрыСервера(
	Порт,
	РазрешенныеOrigins = "",
	ТаймаутСек = 30,
	ИнформацияОСервере = Неопределено) Экспорт
```

```bsl
Функция НоваяИнформацияОСервере(
	Имя,
	Версия,
	Заголовок = "",
	Описание = "",
	Инструкции = "") Экспорт
```

Целевой запуск:

```bsl
Асинх Функция Запустить(ПараметрыСервера) Экспорт
Асинх Функция Остановить() Экспорт
```

## Регистрация через autodiscovery

Целевой контракт найденного провайдера:

```bsl
Процедура Зарегистрировать() Экспорт
```

Внутри runtime:
- определяет `ИсточникРегистрации`;
- вызывает публичные `Зарегистрировать*` методы `Мсп_Сервер`;
- descriptor не знает имя источника.

## Регистрация из внешней обработки

Внешняя обработка регистрирует сущности напрямую через `Мсп_Сервер` и передает строковый `ИсточникРегистрации`.

Пример:

```bsl
Процедура ЗарегистрироватьКомпоненты() Экспорт
	ИсточникРегистрации = "МояВнешняяОбработка";

	Инструмент = Мсп_Сервер.НовыйИнструмент(
		"process_text",
		"Обрабатывает текст",
		СхемаПараметров(),
		Новый ОписаниеОповещения("Обработать", ЭтотОбъект),
		"sync");

	Мсп_Сервер.ЗарегистрироватьИнструмент(Инструмент, ИсточникРегистрации);
КонецПроцедуры
```

Снятие регистрации:

```bsl
Процедура ПередЗакрытием() Экспорт
	Мсп_Сервер.ОтменитьРегистрацию("МояВнешняяОбработка");
КонецПроцедуры
```

## Что меняется относительно текущего ядра

- public API перестает зависеть от `Мсп_ПротоколСлужебный` и `Мсп_Транспорт*Служебный`;
- `Провайдер` уходит из descriptor-контрактов;
- `Асинхронный` уходит из public API;
- task API остается статическим;
- prompts становятся first-class сущностью в том же уровне API;
- `raw` и другие transport-детали исчезают из публичного контракта вызова;
- конфиг сервера становится структурным.

## Осознанные компромиссы

- строковый `ИсточникРегистрации` может конфликтовать;
- этот риск принимается ради простоты реализации в 1С;
- object-handle регистрации не используется;
- `sync_or_task` не поддерживается, чтобы не усложнять ядро и контракт обработчика.
