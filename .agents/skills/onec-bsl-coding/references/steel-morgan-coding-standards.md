# Стандарты кодирования BSL (1С)

## Назначение

Этот навык учит агента писать код на встроенном языке 1С (BSL) в соответствии со стандартами платформы 1С:Предприятие и рекомендациями ИТС. Соблюдение стандартов критически важно: код 1С живёт в продуктиве годами, его читают и поддерживают десятки разработчиков. Единообразный стиль снижает когнитивную нагрузку и количество ошибок.

**Источники стандартов:**
- [Стандарты разработки на ИТС](https://its.1c.ru/db/v8std)
- Соглашения о именовании: ИТС, раздел «Общие требования к конфигурации»
- Стандарты БСП (Библиотека стандартных подсистем)

## Сводка правил

| # | Правило | Обоснование |
|---|---------|-------------|
| 1 | PascalCase на русском | Стандарт ИТС, единообразие |
| 2 | Структура модуля с областями | Навигация, читаемость |
| 3 | Предпочитать &НаСервереБезКонтекста | Снижение трафика клиент-сервер |

| 10 | НСтр() для строк пользователю | Локализация |

| 13 | Избегать Выполнить()/Вычислить() | Безопасность, анализируемость |
| 14 | Нет магическим числам | Читаемость, настраиваемость |

| 19 | Один серверный вызов вместо нескольких | Меньше задержек клиент-сервер |

## Правило 1: Именование переменных — PascalCase на русском языке

**Почему:** Платформа 1С исторически использует русскоязычные идентификаторы в PascalCase. Английские имена допустимы только в экспортных API, предназначенных для внешних систем. Смешение языков в одном модуле затрудняет чтение и поиск.

**Стандарт ИТС:** «Тексты модулей» — имена переменных, процедур и функций должны быть на русском языке, в формате ВерблюжийРегистр.

### Правила именования

| Элемент | Формат | Пример |
|---------|--------|--------|
| Переменная | СуществительноеИлиФраза | `КоличествоСтрок`, `ДатаНачалаПериода` |
| Булева переменная | Утвердительная форма | `ЭтоНовый`, `РазрешеноРедактирование`, `ЕстьОшибки` |
| Параметр | КакПеременная | `ДокументСсылка`, `РежимОткрытия` |

### Правильно

```bsl
Процедура ЗаполнитьТабличнуюЧастьТовары(ДокументОбъект, ДанныеЗаполнения)
    
    КоличествоСтрок = ДанныеЗаполнения.Количество();
    ЕстьОшибки = Ложь;
    
    Для Каждого СтрокаДанных Из ДанныеЗаполнения Цикл
        НоваяСтрока = ДокументОбъект.Товары.Добавить();
        НоваяСтрока.Номенклатура = СтрокаДанных.Номенклатура;
        НоваяСтрока.Количество = СтрокаДанных.Количество;
    КонецЦикла;
    
КонецПроцедуры
```

### Неправильно

```bsl
// Смешение языков, неинформативные имена, нарушение PascalCase
Процедура fill_table(doc, data)
    
    n = data.Количество();
    flag = Ложь;
    
    Для Каждого row Из data Цикл
        nr = doc.Товары.Добавить();
        nr.Номенклатура = row.Номенклатура;
    КонецЦикла;
    
КонецПроцедуры
```

## Именование методов

- Общее правило: Имена процедур берите от неопределенной формы глагола, суть выполняемого действия
- Общее правило: Имена функций берите от описания возвращаемого значения, не используйтие типы
- Функции-конструкторы должны начинаться со слова `Новый`
- Функции-утверждения, выполняющая проверку должны начинаться с `Это|Используется|Включено|Есть` или используй причастие
- Если для понимания назначения функции важно каким образом было получено возвращаемое значение, назовите ее от глагола в неопределенной форме.
- Если функция используется как процедура, ее действие - основная задача, ее результат - побочный эффект, называйте функции как процедуры

## Правило 2: Структура модуля — секции интерфейса и реализации

**Почему:** Чёткая структура модуля позволяет быстро находить нужный код. Экспортные процедуры — это контракт модуля, их размещение вверху позволяет понять API без прочтения всей реализации. Стандарт ИТС описывает конкретный порядок секций.

**Стандарт ИТС:** «Структура модуля» — модуль разбивается на области (#Область), идущие в определённом порядке.

### Порядок секций общего модуля

```bsl
#Область ПрограммныйИнтерфейс

// Экспортные процедуры и функции — публичный API модуля.
// Это «контракт» с вызывающим кодом. Их сигнатуры нельзя менять без анализа всех точек вызова.

Функция ПолучитьКурсВалюты(Валюта, ДатаКурса) Экспорт
    // ...
КонецФункции

#КонецОбласти

#Область СлужебныйПрограммныйИнтерфейс

// Экспортные процедуры, предназначенные для вызова только из других модулей данной подсистемы.
// Не являются публичным API — вызываются только «своими».

Функция ПересчитатьКурсВнутренний(ПараметрыПересчета) Экспорт
    // ...
КонецФункции

#КонецОбласти

#Область СлужебныеПроцедурыИФункции

// Внутренняя реализация. Не экспортные.

Функция СформироватьЗапросКурса(Валюта, Дата)
    // ...
КонецФункции

Процедура ЗаписатьКурсВРегистр(ДанныеКурса)
    // ...
КонецПроцедуры

#КонецОбласти
```

### Порядок секций модуля объекта

```bsl
#Область ОписаниеПеременных

Перем МассивИзменённыхРеквизитов;

#КонецОбласти

#Область ОбработчикиСобытий

Процедура ОбработкаЗаполнения(ДанныеЗаполнения, ТекстЗаполнения, СтандартнаяОбработка)
    // ...
КонецПроцедуры

Процедура ПередЗаписью(Отказ)
    // ...
КонецПроцедуры

#КонецОбласти

#Область СлужебныеПроцедурыИФункции

// Внутренняя логика объекта

#КонецОбласти

#Область Инициализация

// Код инициализации переменных модуля (выполняется при создании объекта)
МассивИзменённыхРеквизитов = Новый Массив;

#КонецОбласти
```

## Правило 3: Директивы компиляции — &НаКлиенте, &НаСервере, &НаСервереБезКонтекста

**Почему:** 1С — клиент-серверная платформа. Выбор директивы определяет, где выполняется код и какие данные передаются между клиентом и сервером. Неправильный выбор директивы приводит к избыточному трафику, медленной работе форм и нарушениям безопасности.

### Описание директив

| Директива | Где выполняется | Доступ к данным формы | Когда использовать |
|-----------|----------------|----------------------|-------------------|
| `&НаКлиенте` | Клиент (тонкий/веб) | Да (клиентская копия) | Интерактивная логика: диалоги, навигация, вызов серверных методов |
| `&НаСервере` | Сервер | Да (полный контекст формы) | Нужен доступ к реквизитам формы + база данных |
| `&НаСервереБезКонтекста` | Сервер | Нет | Запросы к БД, вычисления, не требующие данных формы |
| `&НаКлиентеНаСервереБезКонтекста` | И клиент, и сервер | Нет | Чистые вычисления, форматирование, валидация без БД |

### Предпочтение: &НаСервереБезКонтекста

**Почему предпочитать `&НаСервереБезКонтекста` вместо `&НаСервере`:**

При вызове `&НаСервере` платформа сериализует **весь контекст формы** (все реквизиты, таблицы формы, настройки) и отправляет на сервер, а после выполнения — обратно. Для формы с большой табличной частью (1000+ строк) это могут быть мегабайты данных туда-обратно.

`&НаСервереБезКонтекста` не передаёт контекст формы — только параметры вызова. Это радикально снижает трафик.

### Правильно

```bsl
// Клиент: реакция на действие пользователя
&НаКлиенте
Процедура НоменклатураПриИзменении(Элемент)
    // На клиенте получаем значение из формы и вызываем сервер без контекста
    ДанныеНоменклатуры = ПолучитьДанныеНоменклатуры(Элементы.Товары.ТекущиеДанные.Номенклатура);
    ЗаполнитьСтрокуТоваровНаКлиенте(ДанныеНоменклатуры);
КонецПроцедуры

// Сервер без контекста: обращение к БД, не требует данных формы
&НаСервереБезКонтекста
Функция ПолучитьДанныеНоменклатуры(НоменклатураСсылка)
    
    Возврат Новый Структура("ЕдиницаИзмерения, Цена, СтавкаНДС",
        НоменклатураСсылка.ЕдиницаИзмерения,
        НоменклатураСсылка.Цена,
        НоменклатураСсылка.СтавкаНДС);
    
КонецФункции
```

### Неправильно

```bsl
// ПЛОХО: &НаСервере передаёт весь контекст формы ради одного запроса
&НаСервере
Процедура НоменклатураПриИзмененииНаСервере()
    
    СтрокаТоваров = Объект.Товары.НайтиПоИдентификатору(
        Элементы.Товары.ТекущаяСтрока);
    
    // Ради получения единицы измерения передали всю форму (все табличные части) на сервер и обратно
    СтрокаТоваров.ЕдиницаИзмерения = СтрокаТоваров.Номенклатура.ЕдиницаИзмерения;
    
КонецПроцедуры
```

## Правило 6: Не затеняйте глобальный контекст

**Почему:** Платформа 1С предоставляет глобальные коллекции метаданных: `Документы`, `Справочники`, `Регистры`, `Перечисления` и т.д. Если назвать локальную переменную `Документы`, она скроет глобальный контекст — дальше в коде `Документы` будет ссылаться на локальную переменную, а не на менеджер документов. Это приводит к трудноуловимым ошибкам.

### Правильно

```bsl
// Конкретное имя, не конфликтует с глобальным контекстом
МассивДокументовКОбработке = Новый Массив;
СправочникНоменклатура = Справочники.Номенклатура;
СсылкаНаКонтрагента = Справочники.Контрагенты.НайтиПоКоду("00001");
```

### Неправильно

```bsl
// ПЛОХО: затеняет глобальную коллекцию «Документы»
Документы = Новый Массив;
// Теперь Документы — это массив, а не менеджер документов!
// Документы.РеализацияТоваровУслуг — ошибка!

// ПЛОХО: затеняет «Справочники»
Справочники = ПолучитьСписокСправочников();
```

### Список зарезервированных имён (не использовать для переменных)

`Документы`, `Справочники`, `Регистры`, `Перечисления`, `ПланыОбмена`, `ПланыВидовХарактеристик`, `ПланыВидовРасчета`, `ПланыСчетов`, `БизнесПроцессы`, `Задачи`, `Обработки`, `Отчеты`, `Константы`, `ПараметрыСеанса`, `РегистрыСведений`, `РегистрыНакопления`, `РегистрыБухгалтерии`, `РегистрыРасчета`

## Правило 7: Конкатенация строк — не используйте «+» в циклах

**Почему:** В BSL строки иммутабельны. Каждая операция `Строка1 + Строка2` создаёт новый объект строки в памяти. В цикле из N итераций это O(N²) по памяти и времени (каждая итерация копирует всё предыдущее). Для 10 000 строк разница — секунды vs миллисекунды.

**Стандарт ИТС:** «Эффективная работа со строками».

### Правильно — вариант 1: массив + СтрСоединить()

```bsl
// O(N) по памяти и времени
ЧастиСтроки = Новый Массив;

Для Каждого Элемент Из КоллекцияДанных Цикл
    ЧастиСтроки.Добавить(Элемент.Наименование);
КонецЦикла;

РезультатСтрока = СтрСоединить(ЧастиСтроки, ", ");
```

### Правильно — вариант 2: СтрШаблон() для фиксированного числа подстановок

```bsl
// Для формирования строки с подстановками (до 10 параметров)
ТекстСообщения = СтрШаблон(
    НСтр("ru = 'Документ %1 от %2 на сумму %3 руб.'"),
    НомерДокумента,
    Формат(ДатаДокумента, "ДЛФ=D"),
    Формат(Сумма, "ЧДЦ=2"));
```

### Неправильно

```bsl
// ПЛОХО: O(N²) — каждая итерация создаёт новую строку, копируя предыдущую
Результат = "";
Для Каждого Элемент Из КоллекцияДанных Цикл
    Результат = Результат + Элемент.Наименование + ", ";  // Копирование всей строки на каждом шаге!
КонецЦикла;
```

## Правило 8: Области (#Область) для организации кода

**Почему:** Области позволяют группировать связанный код и сворачивать его в IDE. Это стандартный механизм навигации в модулях 1С. Без областей модуль в 500+ строк становится нечитаемым.

**Стандарт ИТС:** «Структура модуля» — обязательное использование стандартных областей.

### Стандартные области для модуля формы

```bsl
#Область ОписаниеПеременных

#КонецОбласти

#Область ОбработчикиСобытийФормы

&НаСервере
Процедура ПриСозданииНаСервере(Отказ, СтандартнаяОбработка)
    // ...
КонецПроцедуры

#КонецОбласти

#Область ОбработчикиСобытийЭлементовШапкиФормы

// Обработчики событий элементов формы (не табличные части)

#КонецОбласти

#Область ОбработчикиСобытийЭлементовТаблицыФормыТовары

// Обработчики для табличной части «Товары»

#КонецОбласти

#Область ОбработчикиКомандФормы

// Обработчики команд (кнопки формы)

#КонецОбласти

#Область СлужебныеПроцедурыИФункции

// Внутренняя логика модуля формы

#КонецОбласти
```

### Правила для областей

1. **Не создавайте вложенные области** глубже 2 уровней — это запутывает, а не помогает
2. **Не создавайте пустые области** — если в области нет кода, удалите её
3. **Используйте стандартные имена** из ИТС — не придумывайте свои (IDE и инструменты анализа опираются на стандартные имена)

## Правило 9: Комментарии — объясняйте «зачем», а не «что»

**Почему:** Код на BSL читается как естественный язык (русские ключевые слова и идентификаторы). Комментарий «// Увеличиваем счётчик на 1» к строке `Счётчик = Счётчик + 1` — бесполезный шум. Комментарии нужны для:
- Описания **зачем** (бизнес-логика, причина решения)
- Описания **ограничений** и **допущений**
- Описания **публичного API** (экспортные процедуры)

**Стандарт ИТС:** «Описание процедур и функций» — экспортные процедуры и функции ОБЯЗАНЫ иметь комментарий-описание.

### Правильно — описание экспортной функции

```bsl
// Возвращает курс валюты на указанную дату.
// Если на указанную дату курс не установлен, возвращает курс на ближайшую предыдущую дату.
//
// Параметры:
//  Валюта   - СправочникСсылка.Валюты - валюта, курс которой нужно получить.
//  ДатаКурса - Дата - дата, на которую нужен курс. 
//              Если не указана, используется текущая дата сеанса.
//
// Возвращаемое значение:
//  Число - курс валюты. 0 если курс не найден.
//
Функция ПолучитьКурсВалюты(Валюта, ДатаКурса = Неопределено) Экспорт
```

### Правильно — комментарий «зачем»

```bsl
// Сумму округляем до копеек, потому что бухгалтерский учёт не допускает дробных копеек,
// а при пересчёте НДС могут возникнуть дроби.
СуммаНДС = Окр(СуммаБезНДС * СтавкаНДС / 100, 2);
```

### Неправильно

```bsl
// ПЛОХО: комментарий повторяет код
// Присваиваем переменной значение номенклатуры
ТекущаяНоменклатура = СтрокаТоваров.Номенклатура;

// ПЛОХО: комментарий устарел (код изменён, комментарий — нет)
// Получаем курс на начало месяца
КурсВалюты = ПолучитьКурсВалюты(Валюта, ДатаДокумента); // передаём дату документа, не начало месяца!
```

## Правило 10: Используйте НСтр() для строковых литералов

**Почему:** Платформа 1С поддерживает многоязычные конфигурации. `НСтр()` позволяет задать строку на нескольких языках и автоматически выбирает версию для текущего языка интерфейса. Даже если конфигурация одноязычная, использование `НСтр()` — хорошая практика: при будущей локализации не придётся переделывать весь код.

**Стандарт ИТС:** «Использование функции НСтр()» — все строковые литералы, отображаемые пользователю, должны быть обёрнуты в `НСтр()`.

### Правильно

```bsl
// Строка для пользователя — через НСтр()
ТекстПредупреждения = НСтр("ru = 'Документ не может быть проведён. Не заполнена дата.'");

// С параметрами — комбинация НСтр() и СтрШаблон()
ТекстСообщения = СтрШаблон(
    НСтр("ru = 'Остаток товара ""%1"" на складе: %2 %3'"),
    Номенклатура,
    Остаток,
    ЕдиницаИзмерения);
```

### Неправильно

```bsl
// ПЛОХО: строка без НСтр() — не локализуется
ТекстПредупреждения = "Документ не может быть проведён.";

// ПЛОХО: НСтр() внутри цикла с конкатенацией
Для Каждого Строка Из ТаблицаОшибок Цикл
    Результат = Результат + НСтр("ru = 'Ошибка в строке: '") + Строка.Описание;  // НСтр в цикле допустим, но конкатенация — нет
КонецЦикла;
```

## Правило 11: Одна процедура — одна ответственность

**Почему:** Длинные процедуры (100+ строк) трудно читать, тестировать и повторно использовать. Если процедура делает несколько вещей — при изменении одной части легко сломать другую. Разделение на мелкие функции с говорящими именами делает код самодокументируемым.

### Правильно

```bsl
Процедура ОбработкаПроведения(Отказ, РежимПроведения)
    
    ИнициализироватьДанныеДокумента();
    ПроверитьЗаполнениеРеквизитов(Отказ);
    
    Если Не Отказ Тогда
        СформироватьДвижения(Отказ);
    КонецЕсли;
    
КонецПроцедуры
```

### Неправильно

```bsl
// ПЛОХО: одна процедура на 300 строк, делающая всё:
// проверку, формирование движений, пересчёт итогов, отправку уведомлений
Процедура ОбработкаПроведения(Отказ, РежимПроведения)
    // ... 300 строк подряд ...
КонецПроцедуры
```

## Правило 12: Явная типизация параметров в комментариях

**Почему:** BSL — динамически типизированный язык. Без описания типов параметров вызывающий код вынужден читать реализацию, чтобы понять, что передавать. Описание типов в комментарии к экспортной функции — единственный способ документировать контракт.

### Правильно

```bsl
// Создаёт новый элемент справочника «Номенклатура» с заполнением по умолчанию.
//
// Параметры:
//  ДанныеЗаполнения - Структура - содержит поля:
//    * Наименование      - Строка - наименование номенклатуры (обязательно).
//    * ВидНоменклатуры    - ПеречислениеСсылка.ВидыНоменклатуры - вид (обязательно).
//    * ЕдиницаИзмерения  - СправочникСсылка.ЕдиницыИзмерения - ед. изм. Необязательно,

//
// Возвращаемое значение:
//  СправочникСсылка.Номенклатура - ссылка на созданный элемент.
//
Функция СоздатьНоменклатуру(ДанныеЗаполнения) Экспорт
```

## Правило 13: Не используйте «Выполнить()» и «Вычислить()» без крайней необходимости

**Почему:** `Выполнить()` и `Вычислить()` — аналоги `eval()`. Они компилируют и исполняют произвольный код в рантайме. Это:
1. **Угроза безопасности** — если в строку попадут пользовательские данные, возможно выполнение произвольного кода
2. **Невидимость для статического анализа** — средства проверки кода не видят, что находится внутри строки
3. **Трудность отладки** — ошибки внутри `Выполнить()` сложнее диагностировать

### Правильно

```bsl
// Используем прямой вызов по имени через метаданные
МенеджерОбъекта = ОбщегоНазначения.МенеджерОбъектаПоСсылке(СсылкаНаОбъект);
```

### Неправильно

```bsl
// ПЛОХО: динамическое выполнение кода
Выполнить("Результат = Справочники." + ИмяСправочника + ".НайтиПоКоду(Код)");
```

## Правило 14: Магические числа и строки — выносите в параметры

**Почему:** Захардкоженные значения («магические числа») в коде:
1. Непонятны — что означает `Если Статус = 3`?
2. Дублируются — при изменении нужно найти и заменить все вхождения
3. Не настраиваемы — изменение требует правки кода и обновления конфигурации

### Правильно

```bsl
// Используем перечисление — значение самодокументировано
Если Документ.Статус = Перечисления.СтатусыДокументов.Согласован Тогда
    // ...
КонецЕсли;

// Или константу для настраиваемых значений
МаксимальноеКоличествоПопыток = Константы.МаксимальноеКоличествоПопытокОтправки.Получить();
```

### Неправильно

```bsl
// ПЛОХО: что такое 3? Почему именно 5?
Если Документ.Статус = 3 Тогда
    // ...
КонецЕсли;

Для Попытка = 1 По 5 Цикл
    // Откуда взялось 5?
КонецЦикла;
```

## Правило 15: Явные JOIN вместо точечной нотации через ссылки

**Почему:** Длинные цепочки ссылок (`Документ.Контрагент.ГоловнойКонтрагент`) создают неявные JOIN и провоцируют N+1 обращения к базе, особенно в циклах. При составных типах платформа вынуждена делать JOIN ко всем возможным таблицам.

### Правильно

```bsl
// Один запрос с явными JOIN
Запрос = Новый Запрос;
Запрос.Текст =
"ВЫБРАТЬ
|   Товары.Номенклатура КАК Номенклатура,
|   Товары.Номенклатура.ВидНоменклатуры КАК ВидНоменклатуры,
|   Товары.Номенклатура.ЕдиницаИзмерения КАК ЕдиницаИзмерения
|ИЗ
|   Документ.РеализацияТоваровУслуг.Товары КАК Товары
|ГДЕ
|   Товары.Ссылка = &ДокументСсылка";
Запрос.УстановитьПараметр("ДокументСсылка", Документ.Ссылка);
```

### Неправильно

```bsl
Для Каждого СтрокаТоваров Из Документ.Товары Цикл
    ВидНоменклатуры = СтрокаТоваров.Номенклатура.ВидНоменклатуры;
    ЕдиницаИзмерения = СтрокаТоваров.Номенклатура.ЕдиницаИзмерения;
    ИмяКонтрагента = Документ.Контрагент.Наименование;
КонецЦикла;
```

```bsl
Форма = ПолучитьФорму("Обработка.ЗагрузкаДанных.Форма");
Форма.Открыть();
```

## Правило 17: Бизнес-логика не должна жить в модуле формы

**Почему:** Модуль формы отвечает за UI. Бизнес-логика в форме приводит к дублированию, снижает тестируемость и нарушает разделение ответственности. Логику записи и проверок размещайте в модуле объекта.

### Правильно

```bsl
// Модуль объекта документа
Процедура ПередЗаписью(Отказ)
    Для Каждого Строка Из Товары Цикл
        Строка.Сумма = Строка.Количество * Строка.Цена;
    КонецЦикла;
КонецПроцедуры
```

### Неправильно

```bsl
// Модуль формы документа
&НаСервере
Процедура ЗаписатьИПровестиНаСервере()
    ДокументОбъект = РеквизитФормыВЗначение("Объект");
    Для Каждого Строка Из ДокументОбъект.Товары Цикл
        Строка.Сумма = Строка.Количество * Строка.Цена;
    КонецЦикла;
    ДокументОбъект.Записать(РежимЗаписиДокумента.Проведение);
    ЗначениеВРеквизитФормы(ДокументОбъект, "Объект");
КонецПроцедуры
```

## Правило 18: Не проглатывайте исключения

**Почему:** Если исключение только логируется, вызывающий код продолжает работу с ошибочными данными. В транзакциях это приводит к незакрытым изменениям. Либо возвращайте статус, либо пробрасывайте исключение.

### Правильно

```bsl
Попытка
    НачатьТранзакцию();
    // ... запись данных ...
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ЗаписьЖурналаРегистрации("Ошибка", УровеньЖурналаРегистрации.Ошибка,,,
        ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));
    ВызватьИсключение;
КонецПопытки;
```

### Неправильно

```bsl
Попытка
    НачатьТранзакцию();
    // ... запись данных ...
    ЗафиксироватьТранзакцию();
Исключение
    ОтменитьТранзакцию();
    ЗаписьЖурналаРегистрации("Ошибка", УровеньЖурналаРегистрации.Ошибка,,,
        ПодробноеПредставлениеОшибки(ИнформацияОбОшибке()));
    // Ошибка проглочена
КонецПопытки;
```

## Правило 19: Агрегируйте серверные вызовы из формы

**Почему:** Последовательные серверные вызовы при одном действии пользователя дают суммарную задержку. Старайтесь получать все данные за один вызов.

### Правильно

```bsl
&НаКлиенте
Процедура КонтрагентПриИзменении(Элемент)
    ДанныеЗаполнения = ПолучитьДанныеКонтрагента(Объект.Контрагент, Объект.Дата);
    Объект.Договор = ДанныеЗаполнения.Договор;
    Объект.Валюта = ДанныеЗаполнения.Валюта;
    КурсВалюты = ДанныеЗаполнения.Курс;
КонецПроцедуры

&НаСервереБезКонтекста
Функция ПолучитьДанныеКонтрагента(Контрагент, Дата)
    Результат = Новый Структура("Договор, Валюта, Курс");
    Результат.Договор = ПолучитьДоговорПоУмолчанию(Контрагент);
    Результат.Валюта = ОбщегоНазначения.ЗначениеРеквизитаОбъекта(Результат.Договор, "ВалютаВзаиморасчетов");
    Результат.Курс = РаботаСКурсамиВалют.ПолучитьКурсВалюты(Результат.Валюта, Дата);
    Возврат Результат;
КонецФункции
```

### Неправильно

```bsl
&НаКлиенте
Процедура КонтрагентПриИзменении(Элемент)
    ДоговорПоУмолчанию = ПолучитьДоговорПоУмолчанию(Объект.Контрагент);
    ВалютаДоговора = ПолучитьВалютуДоговора(ДоговорПоУмолчанию);
    КурсВалюты = ПолучитьКурсВалюты(ВалютаДоговора, Объект.Дата);
КонецПроцедуры
```
